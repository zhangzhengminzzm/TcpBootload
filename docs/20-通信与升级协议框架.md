# 通信与升级协议框架

## 1. 前提假设

- 设备正常运行时对外提供 `Modbus TCP` 服务。
- 通过特定寄存器/线圈命令触发“进入升级模式”。
- 进入升级模式后可继续复用 Modbus TCP，或切换到 Bootload TCP 私有指令集。

## 2. 推荐协议策略

优先推荐“单通道策略”：升级全流程仍通过 Modbus TCP 承载，优点是网络运维简单、端口统一、调试工具通用。

若设备资源允许，也可采用“控制与传输分离”策略：
- Modbus TCP：负责状态机控制命令
- 私有 TCP：负责大块固件数据传输

## 3. 升级阶段定义

1. `PRECHECK`：设备信息与固件合法性校验
2. `ENTER_BL`：进入 Bootloader 模式
3. `SESSION_START`：会话初始化（版本、分包大小、总包数）
4. `ERASE`：擦除目标 Flash 区域
5. `TRANSFER`：分包下载
6. `VERIFY`：CRC 或签名校验
7. `ACTIVATE`：设置启动标志并重启
8. `POSTCHECK`：应用版本回读确认

## 4. 数据分包建议

- 包大小：`256 ~ 1024 bytes`（根据设备 RAM 与网络稳定性配置）
- 每包字段建议：
  - `session_id`
  - `packet_index`
  - `payload_len`
  - `payload`
  - `packet_crc`
- 响应建议：
  - `ACK(index)` 成功
  - `NACK(index, err_code)` 失败并携带错误码

## 5. 超时与重试建议

- 单包响应超时：`500ms ~ 2000ms`
- 单包重试次数：`3`
- 阶段超时：按阶段设置（擦除可更长）
- 连续失败阈值触发会话中止并回滚到安全状态

## 6. 错误码建议（示例）

- `0x01` 会话无效
- `0x02` 包序号错误
- `0x03` 数据长度非法
- `0x04` CRC 错误
- `0x05` Flash 写入失败
- `0x06` 设备忙
- `0x07` 版本不允许回退

## 7. 与上位机接口约定

上位机需要统一抽象如下接口：

- `connect(ip, port) -> Session`
- `precheck(device_info, fw_meta) -> bool`
- `enter_bootloader() -> bool`
- `transfer_chunk(index, data) -> Ack`
- `verify(image_crc) -> bool`
- `activate_and_reboot() -> bool`
- `query_version() -> str`
